# v0.33 Storage UX And Capability Hardening

## Purpose

v0.33 is a local-first confidence release. It should make Artifact's browser
storage feel understandable and recoverable by showing autosave, recovery,
quota, project-size, cleanup, and browser capability states without changing
the document schema, renderer/export semantics, graph traversal, AI scope,
server sharing, or account model.

This release exists because Artifact already depends on localStorage,
IndexedDB, imported asset references, local projects, recovery copies, and
browser-only APIs. v0.33 should make those constraints visible enough that a
user can trust the editor before the larger Asset Library, Export History,
Versions, and server-backed sharing releases build on the same storage model.

## Product Direction

Artifact should continue to feel local-first instead of cloud-first. Users
should know when their work is saved, when a recovery path exists, when a
project is getting large, and when their browser cannot support an expected
capability. The UI should provide practical status and recovery actions, not
internal implementation language.

The release should answer:

- How does the editor show that current work has a saved local snapshot without
  claiming a durable open-project binding?
- How does a user restore a recovery copy or understand that recovery is
  available?
- How does the app warn before local storage limits become destructive?
- How does a user see project size and clean up unused local assets safely?
- How does the app handle unsupported WebGL, storage, download, or file APIs?
- How much PWA support helps local-first trust without creating stale-cache
  deploy risk?
- Which browser-gate stability issues are test-harness problems versus product
  capability states?

## Non-Goals

- No server-backed share links or cloud project storage.
- No account sync, collaboration, billing, or backend product work.
- No broad Asset Library, Export History, or Versions feature yet.
- No push notifications, background sync, file handlers, or desktop-app shell
  claims for the first PWA slice.
- No document schema change unless a focused migration is required for a
  storage-safety bug and is covered by tests.
- No renderer/export, graph traversal, package export, AI scope, or font-policy
  semantic changes.
- No landing refresh, Showcase / How-to split, command palette, or Add Library
  feature work.
- No broad redesign of the editor chrome.
- No automatic deletion of local assets without a visible user action and a
  safe recovery or confirmation path.
- No active project save model yet. v0.33 may show when the current document
  matches a snapshot saved in the current session, but the Projects save action
  still creates a local snapshot rather than overwriting a durable active
  project id.

## Workstreams

### 1. Autosave And Recovery Visibility

- [x] Audit current active-document persistence, local project save, imported
  asset storage, and pre-blank recovery copy flows.
- [x] Add user-facing autosave status that reflects local persistence without
  exposing implementation details such as localStorage or IndexedDB names.
- [x] Add recovery visibility for available drafts and interrupted starts.
- [x] Keep browser storage and offline-app diagnostics out of the primary
  Projects view; expose them only behind `Storage details`.
- [x] Remove the separate local Presets sheet so Projects stays the only local
  snapshot surface until true templates or style presets have a distinct model.
- [x] Label the Projects save action as snapshot saving so current UI copy
  matches the snapshot-only persistence model.
- [x] Keep destructive starts guarded by confirmation and recovery behavior.
- [x] Add focused tests for persistence status and recovery-state transitions.

### 2. Project Size, Quota Pressure, And Cleanup

- [x] Estimate and display local project size or storage pressure in the
  smallest useful way.
- [x] Detect quota errors and show a clear recovery path instead of failing
  silently.
- [x] Identify unused imported assets where this can be proven from local
  project references. v0.33 does not add broad asset garbage collection because
  the current traceable cleanup targets are saved projects and the recoverable
  draft; unused asset GC remains part of a future Asset Library / cleanup pass.
- [x] Add cleanup actions only when the target assets are traceable and the
  user can understand what will be removed. Existing project delete and
  recovery-draft delete remain the safe cleanup actions in this release.
- [x] Preserve imported image/font portability for `.artifact.json` and project
  package flows.

### 3. Browser Capability Hardening

- [x] Audit WebGL, Canvas, IndexedDB, localStorage, file input, download, and
  worker capability checks used by the editor.
- [x] Add or improve capability fallbacks for unsupported browser APIs where a
  practical fallback exists.
- [x] Show concise capability warnings when a browser cannot support an
  expected editor or export flow.
- [x] Keep warning copy product-facing and action-oriented, not agent or QA
  language.

### 4. PWA Install And Offline Shell

- [x] Add a conservative web app manifest with install metadata, app name,
  theme color, and start URL.
- [x] Register a production-only service worker so dev and local browser tests
  do not accidentally serve stale bundles.
- [x] Cache only the app shell/static assets and navigation fallback; do not
  cache user documents, API responses, AI requests, or share payloads as hidden
  product state.
- [x] Add an offline fallback page for navigations that cannot be served from
  the cached shell.
- [x] Cover manifest and service-worker asset availability in browser tests.

### 5. Local Project And Asset Store Resilience

- [x] Review `assetStore`, `projectStore`, document persistence, URL import,
  and project package flows for recoverable failure paths.
- [x] Ensure failed asset imports, missing assets, and blocked storage writes
  leave the editor usable.
- [x] Keep decoded image, render, thumbnail, Pixi, Three.js, and worker caches
  outside serialized document state.
- [x] Add focused tests around serialization-safe storage behavior before any
  persistence refactor.

### 6. Browser Gate And Release Harness Stability

- [x] Document the local Playwright `reuseExistingServer` behavior and how to
  avoid stale dev-server release-gate runs.
- [x] Decide whether local browser tests should keep reusing existing servers
  by default or use a release-gate mode that always starts fresh.
- [x] Keep CI browser checks blocking; do not weaken browser coverage because
  of local harness cleanup.
- [x] Add the smallest script or doc update needed so future release prep does
  not confuse stale local servers with product failures.

### 7. Docs And Release Hygiene

- [x] Update `docs/state-model.md`, `docs/testing.md`, and
  `docs/production-readiness.md` when storage or release-gate behavior changes.
- [x] Keep internal QA notes out of user-facing app surfaces.
- [x] Create `docs/releases/v0.33.0.md` from `docs/release-template.md` during
  release prep.

## Acceptance Criteria

- The editor clearly communicates local save/recovery state for active work.
- Recovery copies are discoverable and safe to restore or dismiss.
- Quota/storage failures produce a visible, actionable state instead of silent
  data loss.
- Project size or storage pressure is visible enough to guide cleanup.
- Cleanup actions only remove assets that are traceably unused or explicitly
  selected by the user.
- Unsupported browser capabilities produce clear fallback or warning states.
- Local project, asset, import/export, and recovery flows remain serializable
  and do not store browser objects inside `CanvasDocument`.
- `.artifact.json` and local project behavior remain compatible with existing
  documents.
- Browser release-gate behavior is documented or scripted so stale local
  servers do not masquerade as product failures.
- Server-backed sharing, broad asset library, export history, and project
  versions remain deferred out of v0.33.
- Active project id binding, overwrite semantics for `Save`, and explicit
  `Save as` / duplicate project behavior remain deferred out of v0.33.

  behavior.

## Suggested Implementation Order

1. Audit current persistence, asset, project, and recovery flows.
2. Define the user-facing storage states and where they should appear.
3. Add pure helpers and tests for storage status, quota classification, and
   capability detection.
4. Wire the smallest UI surfaces for autosave, recovery, and quota pressure.
5. Add safe cleanup only after unused-asset detection is traceable.
6. Stabilize/document local browser release-gate server behavior.
7. Update architecture/testing/production-readiness docs.
8. Run the full release gate and create template-backed release notes.
