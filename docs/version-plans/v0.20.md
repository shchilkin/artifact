# v0.20 Text Workflow v1

## Summary

v0.20 should make text feel like a first-class creative workflow rather than a
single generic layer control. The release should help users build cover-ready
typography faster: stronger starts, clearer font choices, better text controls,
and parity between Layers and Nodes.

The first slice is **Text Workflow v1**.

## Goals

- Make adding text feel intentional: title, subtitle, label, credit, and poster
  starts should be easy to reach.
- Improve font browsing without turning the editor into a corporate typography
  admin panel.
- Make text controls read as creative handles: size, tracking, line height,
  width/wrap, alignment, rotation, opacity, and blend should be clearer and more
  useful.
- Keep Layers and Nodes consistent through the shared text layer model and
  existing inspector surfaces.
- Add a small set of text-oriented starter/example documents that show
  multi-font cover workflows.
- Preserve render/export parity for text across preview, node thumbnails,
  graph output, examples, and export.

## Non-Goals

- No custom font upload in v1.
- No external font catalog integration in v1.
- No new durable `CanvasDocument` schema fields unless an existing text control
  cannot express the intended workflow.
- No renderer rewrite and no change to graph traversal, thumbnails, or export
  semantics.
- No command palette work.
- No backend/accounts/project-sync work.

## Acceptance Criteria

- Users can add useful text starts without manually engineering every value.
- Font choices are easier to scan and understand.
- Text controls feel like creative cover-design controls, not raw debugging
  fields.
- Layers and Nodes expose the same text behavior.
- Preview, graph output, thumbnails, and export remain consistent.
- No renderer/export semantics change is introduced.
- No schema migration is required unless explicitly justified and tested.
