# v0.29 Product Surface Recovery

## Summary

v0.29 should recover the public product surface around Artifact without trying
to solve the landing page in the same slice. The current product is stronger
than the public surface: Layers, Nodes, typography, imported fonts, packages,
Add Library, and guardrails already make Artifact feel like a real local-first
creative editor, but the old examples/docs/terminology surfaces made that hard
to understand.

Important status update: **landing refresh is no longer part of v0.29**. The
landing explorations remain useful reference material, but they are explicitly
deferred. v0.29 should close as a product-surface cleanup around terminology,
showcase, docs, shared UI primitives, and validation coverage.

The v0.29 direction remains **Structured Canvas, Open Outcomes**: clear enough
to trust, tactile enough to remember, and product-focused enough to understand.
The page should feel like a poster editor with a print-proof interface language,
not like a generic SaaS landing and not like an internal moodboard.

This slice is about product understanding and visual confidence, not backend
expansion. The goal is that a new user can understand that Artifact is an
editor, browse made-in-Artifact work, find task-oriented docs, open a blank
editor by default, and trust that typography, effects, local projects, packages,
and export are first-class workflows.

## Product Positioning

Artifact is not a generic AI image generator, template tool, moodboard app, or
inspiration board. It is a **local-first creative editor** for covers, posters,
type compositions, textures, effects, and export-ready visual work.

The central product thesis:

> Same document. Two ways to build.

- **Layers mode** is the fast path: familiar stack composition, direct controls,
  quick edits, and export without breaking the flow.
- **Nodes mode** is the deeper path: graph composition, visible chains,
  source/effect/merge/output control, and explicit output targeting.

Future landing work should show both as equally real product surfaces. Nodes
should not look like a hidden advanced feature; they are the deeper path after
fast layer-based creation.

## Goals

- Stabilize public terminology around Artifact as an editor, not a generator.
- Make public navigation clear: Docs, Showcase, Open editor, and blank-canvas
  default behavior.
- Split showcase and learning paths into separate surfaces:
  - **Showcase**: work made in Artifact, starting with curated projects plus
    seeded random exploration, with future reviewed agent-generated entries.
  - **How-to / recipes**: a future task-oriented learning surface for doing
    specific things in Artifact.
- Replace the stale public examples surface with a clearer showcase gallery
  that demonstrates what Artifact can make without overwhelming the user with
  process text.
- Move workflow teaching into future how-to / recipes surfaces so a user can
  learn intentionally instead of decoding a mixed gallery/manual page.
- Audit the current docs information architecture. The docs exist, but the
  current `/docs/nodes` surface mixes onboarding, recipes, reference catalog,
  troubleshooting, and live node demos in one long page, which may be why it
  feels useful but not satisfying.
- Add a small visual confidence baseline for the product surfaces most likely
  to drift.

## Non-Goals

- No server-backed share links.
- No account sync or project sync.
- No AI expansion.
- No renderer, export, graph traversal, thumbnail scheduling, font package
  policy, or document schema changes.
- No corporate SaaS/admin visual redesign.
- No landing redesign or landing release-quality acceptance in v0.29.
- No promise of unlimited cloud share, collaboration, account continuity,
  community galleries, or server-side project storage.

## Deferred Landing Direction

The following landing direction is preserved as reference for a future focused
landing pass. It is not a v0.29 acceptance requirement.

Use modern creative-workspace discipline, then make it Artifact:

- large confident hero copy;
- dark editorial canvas and subtle grid;
- floating product-proof cards instead of decorative cards;
- hard panels, crop marks, registration marks, mono labels, and muted borders;
- off-white text with artifact red/hot-pink active accents;
- restrained violet glow only where it helps graph/effect readability;
- real workflow fragments: poster output, layer stack crop, node graph crop,
  effect controls, font specimen, and export/package card.

Avoid generic startup gradients, glossy AI SaaS styling, 3D blobs, pastel
collaboration vibes, lifestyle-only galleries, and pure output grids that hide
the editor.

### Selected Reference Direction: Specimen Wall / Workflow Gallery

The chosen landing direction is a **Specimen Wall / Workflow Gallery**, not a
generic feature page and not a plain "how it works" strip. The page should feel
like a print proof wall: large irregular Artifact outputs, tiny process labels,
and embedded product surfaces that reveal how the work was made.

The selected content spine is:

> Start -> Shape -> Route -> Finish -> Reuse

This is the page's information architecture. Keep user-facing copy on that
spine and avoid internal implementation language like "renderer proof",
"feature catalog", or "final build". The first headline should sell capability
without prescribing taste:

> Make cover art you can keep shaping.

Supporting copy should carry editability and product depth:

> Start with an image, type, fill, or procedural source. Work fast in Layers,
> route deeper in Nodes, and export a clean raster when the piece is ready.

The useful pattern is:

- one dominant editorial statement;
- an irregular wall of outputs and product-proof snippets;
- annotations like `LAYERS`, `NODES`, `TYPE`, `EXPORT`, and `EFFECT CHAIN`;
- real surfaces embedded between outputs: layer rows, node graph, font card,
  package/export card, and renderer-backed effect specimens;
- showcase presented as outcomes, while guides/docs carry learning.

For Artifact, adapt this as a **workflow gallery**:

1. **Outputs first** — show posters/covers/type artifacts as the main emotional
   proof.
2. **Process in the wall** — place layer stack, node graph, effect controls,
   font, and package panels inside the wall so visitors see the editor behind
   the work.
3. **Same document, two ways to build** — use a later section to clarify Layers
   and Nodes without making them feel like unrelated feature cards.
4. **Effects must be real** — effect specimens should use the renderer path or
   honest product captures, not decorative CSS placeholders.
5. **Split showcase and guides** — showcase stays gallery/remix oriented;
   guides and docs explain workflow.

Visual treatment:

- Use a dark, print-like, mono, low-chrome identity with hard rectangular
  surfaces.
- Keep the wall asymmetrical and specimen-like, not a tidy SaaS card grid.
- Product proof snippets can be stylized, but they must read as editor
  surfaces rather than decorative illustrations.
- Motion can be a subtle proof-wall drift only; it must respect reduced motion
  and remain readable when static.
- The first screen should make a visitor think "this is a tool for making this
  kind of visual work", not "this is a product feature list".

## Deferred Landing Structure

This structure is not part of v0.29. Use it only when the landing work is
reopened as a dedicated slice.

### 1. Hero

Purpose: communicate the product instantly.

Recommended headline:

> A local-first creative editor for cover art you can keep shaping.

Supporting copy:

> Start fast with layers. Go deeper with nodes. Build textured artwork with
> editable type, procedural sources, effect chains, local projects, and clean
> raster export.

Primary CTA: **Open Artifact**.

Secondary CTA: **View Showcase**. Future how-to / recipes can become a
separate learning path when that surface exists.

Hero visual requirements:

- central editorial copy over a dark/grid canvas;
- floating cards mixing final artwork and product UI fragments;
- cards prove real surfaces: poster output, layer stack, node graph, effect
  controls, font specimen, export/package state;
- no purely decorative hero image.

### 2. Same Document, Two Ways To Build

Purpose: make Layers vs Nodes obvious.

Layers copy should show:

- image, text, fill, emoji, noise, primitive, and effect layers;
- reorder, visibility, lock, rename, duplicate, delete, and export;
- fast direct composition.

Nodes copy should show:

- source, effect, merge, and output nodes;
- connected branches and graph composition;
- drag-and-drop insertion into graph paths;
- output target and preview/export trust.

Key line:

> Same document. Two mental models.

This section can use the fast visual walkthrough pattern: a single artwork or
sequence of artifacts should visibly pass through Layers and Nodes rather than
describing them as two unrelated feature cards.

### 3. Effects Made For Editable Material

Purpose: sell creative controls as artistic tools.

Show effect specimens for Grain, Dither, Halftone, Pixelate, Scanlines,
Misregister, Glow, Chromatic Aberration, Warp, and low-res sampling. Use
before/after strips, routed effect chains, or a specimen grid with labels like
`GRAIN`, `DITHER`, `MISREGISTER`, and `OUTPUT`.

The copy should emphasize that controls are tuned for artwork, not engineering
numbers.

### 4. Typography Stays Editable

Purpose: make Artifact feel mature rather than toy-like.

Show editable text layers, Font Library, local font import, Google Fonts import,
font metadata, license-aware package policy, and raster export safety.

Core message:

> Use built-in fonts, import local fonts, or bring in Google Fonts. Text remains
> editable in the project, while raster exports do not silently distribute font
> files.

### 5. Local-First Projects

Purpose: frame portability as strength.

Show `.artifact`, `.artifact.json`, local open/import, raster export, package
export, and lightweight share links. Be honest without sounding defensive:

> Save local projects, open them later, export raster artwork, or package the
> project when you need to move it. Lightweight projects can be shared by link;
> larger project continuity is a future surface.

### 6. Showcase And How-To

Purpose: avoid mixing two different jobs on one page.

Showcase should be a gallery surface at `/showcase`:

- curated made-in-Artifact work first;
- seeded random work second;
- future agent-generated work only after a reviewed pipeline exists;
- an infinite wall of images first, with no explanatory product copy between the
  header and the wall;
- filters/tags only as a future affordance once the gallery is large enough to
  need them;
- concise tiles that show output, title, type, and entry action;
- no long workflow explanations inside the gallery grid.

How-to / recipes should become the future learning surface:

- task-oriented flows such as "Start with Layers", "Go deeper with Nodes",
  "Build a type cover", "Export/package safely";
- a small sequence of steps;
- links to open a starter or showcase document;
- enough explanation to teach without turning the showcase gallery into a docs
  page;
- clear separation from reference docs.

Original target workflows remain useful, but they belong in guides/flows rather
than being forced into every gallery card.

Each showcase project should show final output, tiny process hint, and tags such as
`LAYERS`, `NODES`, `TYPE`, `EXPORT`, and `TEXTURE`.

Target showcase projects:

- **Clean type cover** — big text, image, grain, misregister, export.
- **Textured poster recipe** — image stack, halftone, scanlines, chromatic
  aberration.
- **Node composition chain** — Source -> Noise -> Dither -> Glow -> Output.
- **Multi-font type system** — import font, build editable text composition, export
  raster safely.
- **Photo cover with object branch** — typography, image, texture, primitive shapes,
  package/export.

### 7. Final CTA

Use a closing line that keeps the product specific:

- **Open the editor and make a poster.**
- **Start with layers. Go deeper with nodes.**
- **Shape the cover. Keep the project yours.**

Primary public CTA: **Open editor**, opening a blank canvas by default.
**New blank canvas** has the same default behavior. Showcase tiles are the path
into existing editable projects.

## Visual And UX Rules

- Keep high-contrast text and clear keyboard focus states.
- Do not rely on color alone for meaning; pair colors with labels/status text.
- Respect reduced-motion preferences.
- Keep mobile readable; floating cards must not obscure copy on small screens.
- Use hard or slightly rounded rectangles, editor panels, crop marks, frame
  outlines, graph edges, layer rows, and technical badges.
- Avoid soft SaaS bubbles, generic gradients, abstract blobs, and pure
  decoration.
- Motion is optional and should describe workflow: node edge drawing, layer
  reorder, before/after scrub, output preview update, or subtle scanline/noise.

### Random Showcase v2 Follow-Up

The current random showcase entries are useful for fast exploration, but they are too
generic as product proof. The algorithm mostly combines an emoji/procedural
source, one or two text layers, and a shuffled effect stack. That means many
outputs read as variations of one idea.

v2 should introduce distinct random archetypes:

- **Image-backed poster** — source image or bundled starter image, type,
  texture, and print finish.
- **Type-only poster** — large editable type, small metadata labels, restrained
  texture, export-ready composition.
- **Texture/source study** — procedural source, paper/noise/array systems, low
  or no typography.
- **Texture/effect stack** — halftone, scanlines, misregistration, grain, and
  controlled degradation.
- **Node-ready composition** — random document that can map cleanly into a graph
  recipe, not only a flat stack.

Potential inputs for v2:

- curated text packs by mood/use case: album, club poster, archive label,
  signal/noise, fashion, horror print, ambient, industrial;
- bundled source-image library if we decide the bundle-size/provenance tradeoff
  is worth it;
- random profiles with weighted layer/effect choices, rather than one global
  shuffle.

The first v2 slice now covers the first four archetypes in code and routes
seeded random showcase entries through the same formula system. Follow-up work
can add node-ready compositions, richer text packs, a bundled source-image
library, and reviewed agent-generated entries.
