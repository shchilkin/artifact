# v0.22 Project & Asset Robustness

## Summary

v0.22 should harden the local-first project story after the v0.20 text workflow
and v0.21 local font import releases.

The goal is confidence: a user should be able to build a document with imported
images, imported fonts, layers, nodes, and export settings, then save, reopen,
share, reload, and export without wondering whether the composition silently
lost an asset.

The first slice is **Project & Asset Robustness v1**.

## Goals

- Make the document's external dependencies explicit: imported images, imported
  fonts, graph output target, export config, and portable payloads.
- Consolidate portable document preparation so save, open, copy-link, local
  project, and future bundle flows do not drift.
- Improve missing asset/font fallback states so the editor stays usable and
  explains the problem without exposing implementation details.
- Strengthen preview/export/share confidence with regression tests that combine
  imported image and imported font cases.
- Preserve render semantics, graph traversal, thumbnail scheduling, document
  history, and local-first persistence boundaries.

## Non-Goals

- No backend storage.
- No account-synced project library.
- No zip/custom `.artifact` project bundle UI in v1.
- No renderer rewrite.
- No graph traversal or node kind change.
- No new durable binary payloads in active `CanvasDocument` state.
- No large project management screen or corporate-feeling asset manager.

## Architecture Notes

- `CanvasDocument` remains the serialized creative model. It may carry portable
  payloads during import/export boundaries, but active editor state should keep
  lightweight `artifact-asset://...` and `artifact-font://...` references.
- Image bytes live in `apps/web/app/utils/assetStore.ts`.
- Font bytes live in `apps/web/app/utils/fontStore.ts`.
- Local projects and recovery draft live in
  `apps/web/app/utils/projectStore.ts`.
- Hydration and stripping of portable payloads should be centralized behind
  named helpers instead of being reassembled independently by each UI surface.
- Missing payloads should degrade to visible, actionable UI states. Renderer and
  export paths must avoid blanking the whole composition because one optional
  local dependency is unavailable.

## Acceptance Criteria

- Save/open/share/export flows use a common portable-document boundary instead
  of duplicating asset/font hydration logic in UI components.
- Active editor documents stay lightweight and serializable.
- Imported image and imported font documents survive save/open and share-link
  round trips where browser limits allow.
- Missing local assets produce understandable editor states instead of blank or
  technical failure states.
- Preview/export behavior remains unchanged for documents with all assets
  present.
- Graph traversal, node thumbnail scheduling, and renderer semantics are not
  changed.
- Release notes call out any accepted limits around very large portable
  payloads or browser URL size.
