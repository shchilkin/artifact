# v0.13 Backend Contract

This contract defines the first private AI generation backend for Artifact. The
frontend is the Vercel React Router app; the generation API, queue, worker,
database, provider keys, and generated file storage live on the VPS.

## Repository Layout

The backend lives in the same repository under `apps/api`, while the existing
React Router/Vercel app remains at the root-level `app/` directory.

```text
apps/api
  README.md
  .env.example
  package.json
  tsconfig.json
  src/
    contracts.ts       Endpoint paths and shared backend-facing contract types
    config.ts          Environment parsing
    server.ts          VPS API process entry point
    worker.ts          Generation worker process entry point
    db/                Migrations and repositories
    providers/         OpenAI/xAI/mock provider adapters
    storage/           Local or object-storage adapters
```

This is intentionally not an npm workspace yet. Root Vercel build/install
behavior should keep serving the existing frontend until backend deployment is
explicitly configured.

## Service Boundary

Browser/Vercel app responsibilities:

- Show login/access/quota state.
- Create generation jobs through the VPS API.
- Poll job status.
- Download completed generated assets through an authenticated asset URL.
- Store the downloaded image in local IndexedDB through the existing
  `artifact-asset://...` flow.
- Insert the generated image into the document as a normal image source.

VPS responsibilities:

- Own auth verification and user entitlements.
- Own provider secrets.
- Enforce quotas, rate limits, idempotency, and concurrency limits.
- Create durable generation jobs.
- Run async workers and provider calls.
- Store generated files and asset metadata.
- Expose job and generated-asset read APIs.
- Keep logs, cost/usage records, and admin diagnostics.

## Auth And Access

The first alpha can use any existing auth infrastructure, but the API contract
expects authenticated browser requests to include credentials/cookies or a
server-issued token accepted by the VPS API.

User access is controlled by a server-side entitlement:

- `ai_enabled`: private alpha access flag.
- `role`: optional admin/operator support.
- `plus_status`: optional future product entitlement.

Anonymous users and logged-in users without access must be able to read access
state, but cannot create generation jobs.

## API Endpoints

All endpoints return JSON unless serving image bytes. Error responses should use:

```json
{
  "code": "quota_exceeded",
  "message": "Monthly quota used."
}
```

### GET /api/ai/access

Returns the current user's AI access state. Anonymous users receive a `200`
response with `authenticated: false` so the frontend can show login copy.

Response:

```json
{
  "authenticated": true,
  "enabled": true,
  "user": {
    "id": "user_123",
    "email": "name@example.com",
    "role": "admin"
  },
  "quota": {
    "period": "2026-05",
    "limit": 10,
    "used": 2,
    "remaining": 8
  },
  "providers": ["openai", "xai"]
}
```

Disabled response:

```json
{
  "authenticated": true,
  "enabled": false,
  "disabledReason": "not_enabled",
  "quota": {
    "period": "2026-05",
    "limit": 0,
    "used": 0,
    "remaining": 0
  },
  "providers": []
}
```

### POST /api/ai/generations

Creates or returns an existing idempotent generation job.

Request:

```json
{
  "prompt": "grainy shoegaze album cover, scanned print texture",
  "provider": "openai",
  "model": "gpt-image",
  "settings": {
    "aspect": "1:1",
    "quality": "standard",
    "stylePreset": "shoegaze",
    "negativePrompt": "text artifacts",
    "sourceAssetId": "optional-server-asset-id"
  },
  "idempotencyKey": "client-generated-uuid"
}
```

Response `201` for a new job, `200` for an existing idempotent job:

```json
{
  "id": "job_123",
  "status": "queued",
  "provider": "openai",
  "model": "gpt-image",
  "prompt": "grainy shoegaze album cover, scanned print texture",
  "settings": {
    "aspect": "1:1",
    "quality": "standard",
    "stylePreset": "shoegaze"
  },
  "quota": {
    "period": "2026-05",
    "limit": 10,
    "used": 3,
    "remaining": 7
  },
  "createdAt": "2026-05-20T10:00:00.000Z"
}
```

Rules:

- Requires authenticated `ai_enabled` user.
- Enforces per-user monthly quota before enqueue.
- Enforces one active job per user for the private alpha.
- Enforces per-user and per-IP rate limits.
- Uses `(user_id, idempotency_key)` to prevent duplicate paid jobs.
- Stores prompt/settings server-side before enqueueing.

### POST /api/ai/generations/:id/cancel

Cancels a queued job, or marks a running job as cancellation requested if the
provider cannot stop work immediately.

Response: same job shape as `GET /api/ai/generations/:id`.

Rules:

- Owner or admin only.
- Cancellation should not refund quota automatically unless no provider call was
  started.

### GET /api/assets/:id/file

Returns image bytes for an authenticated owner or admin. This endpoint is what
the Vercel app downloads before storing the image locally with
`saveImageBlobAsset(...)`.

Rules:

- Owner or admin only.
- Return cache headers carefully. Private alpha can use `private, max-age=300`.
- Do not expose raw storage keys to the browser.

## Job Lifecycle

```text
queued -> running -> succeeded
queued -> cancelled
queued -> expired
running -> succeeded
running -> failed
running -> cancelled
failed -> queued        only through explicit admin/user retry
```

Quota behavior:

- Creating a job reserves or consumes one generation.
- If a job fails before a provider request starts, quota may be restored.
- If the provider request starts, quota remains consumed by default.
- Idempotent duplicate requests must not consume extra quota.

Retry behavior:

- Provider/network transient errors can retry with bounded backoff.
- Policy, auth, quota, invalid settings, and unsupported provider errors are not
  retryable.
- After the retry limit, mark the job `failed` and keep the error inspectable.

## Queue And Worker

BullMQ plus Redis is the preferred first implementation.

Queue payload should be small:

```json
{
  "jobId": "job_123",
  "userId": "user_123"
}
```

The worker should load the full job from the DB before calling the provider.
Never put prompt history, image bytes, or provider secrets in Redis payloads.

Worker steps:

1. Load job and verify it is still `queued`.
2. Mark `running`, increment `attempt_count`, set `started_at`.
3. Call provider adapter.
4. Validate output mime/size/dimensions.
5. Write image bytes to storage.
6. Create `assets` record.
7. Mark job `succeeded` with `output_asset_id`.
8. On failure, mark `failed` with `error_code`, `error_message`, and
   `retryable`.

## Rate Limits And Concurrency

Minimum private-alpha limits:

- Monthly quota: 10 successful/provider-started generations per user.
- Active jobs: 1 queued/running job per user.
- Create endpoint: short-window per-user and per-IP limit.
- Asset download: per-user and per-IP limit to avoid hot-loop downloads.

Redis is appropriate for short-window counters. DB remains the source of truth
for monthly quota usage.
