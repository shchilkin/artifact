# v0.21 Font Import / Typography Library v2

## Summary

v0.21 should continue the typography direction opened by v0.20. The goal is to
let users bring their own type into Artifact without turning the editor into a
large font-management app or changing document/render semantics.

The first slice is **Font Import / Typography Library v2**.

## Goals

- Let users import local font files for browser-only editing:
  `.ttf`, `.otf`, `.woff`, and `.woff2`.
- Keep imported fonts available in the Font Library under a clear imported/local
  section.
- Preserve Layers and Nodes parity through the existing shared font picker and
  text controls.
- Keep `CanvasDocument` lightweight: text layers should reference a font id or
  font family string, not store font binary payloads.
- Store imported font payloads outside the document, using the same local-first
  persistence principles as image assets.
- Make preview, node thumbnails, graph output, and export wait for custom font
  readiness before drawing text when the browser supports the relevant APIs.
- Make `.artifact.json` behavior explicit and tested: imported fonts should be
  hydrated into portable exports where possible, with deterministic fallback
  behavior where not possible.

## Non-Goals

- No backend font storage.
- No account-synced font sets.
- No arbitrary full Google Fonts catalog browsing in v1.
- No text-on-path, text masks, or advanced layout tools.
- No new graph node kind.
- No renderer rewrite, graph traversal change, thumbnail scheduling change, or
  export pipeline rewrite.
- No durable schema migration unless the existing text font field cannot safely
  identify imported fonts.

## Architecture Notes

- Local font payloads should live outside `CanvasDocument`, likely through a new
  font store or an asset-store extension that can handle font MIME types.
- `CanvasDocument` should store only a serializable reference such as an
  `artifact-font://...` id or a normalized custom font id.
- The shared font registry in `apps/web/app/types/typography.ts` remains the
  source for bundled/curated fonts. Imported fonts should compose with it rather
  than replacing it.
- Font loading should remain a render input/cache concern. Do not store
  `FontFace`, `ArrayBuffer`, DOM objects, or other non-serializable values in
  document state.
- `.artifact.json` export and share-link creation should hydrate local font
  references where feasible, similar to image asset hydration.
- Imported font deletion/removal must not corrupt documents that still reference
  the font. The UI should fall back visibly and predictably to a bundled font.

## Acceptance Criteria

- Users can bring at least one local font file into the editor and use it in
  real cover work.
- Imported fonts appear in the same Font Library experience as bundled fonts,
  without a separate corporate-feeling management screen.
- Layers and Nodes expose the same imported font behavior.
- Preview, node thumbnails, graph output, and export remain consistent.
- `CanvasDocument` remains serializable and lightweight.
- `.artifact.json` behavior for imported fonts is documented and covered.
- No renderer/export semantics change is introduced beyond waiting for the
  selected font to be ready.
- No graph traversal, thumbnail scheduling, or AI scope change is introduced.
