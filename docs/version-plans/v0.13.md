# v0.13 Acceptance Plan

## Theme

AI Generation Research And Architecture.

The goal of v0.13 is to add a private, quota-limited AI image generation alpha
without weakening Artifact's local-first editor, leaking provider secrets into
the browser, or creating unbounded generation costs.

## Product Shape

- AI generation is available only to authenticated users with `ai_enabled` or a
  plus/admin equivalent.
- The first user-facing workflow is an Image Generation node that creates a
  normal image asset which can flow through existing effects, merge nodes,
  thumbnails, preview, and export.
- OpenAI GPT Image is the primary provider. xAI image generation can be a
  secondary provider for alternate creative output, still behind Artifact's own
  policy, quota, and moderation rules.
- Generation runs through a VPS backend, not directly from the Vercel app or
  browser.
- Generated outputs become Artifact assets with metadata, not a separate AI-only
  document path.

## Architecture Direction

- Frontend: React Router app remains deployed on Vercel from the
  `apps/web` workspace.
- Backend: VPS API owns auth verification, quotas, rate limits, provider
  secrets, generation job creation, and asset records.
- Worker: async generation worker processes jobs from a queue, calls providers,
  stores outputs, and updates job state.
- Queue: BullMQ plus Redis is the preferred first implementation if the VPS
  already owns the infrastructure.
- Database: stores users, generation jobs, quota usage, assets, and future
  project ownership.
- Storage: generated images should live in file/object storage with DB metadata.
  Browser IndexedDB can import the finished asset for local editing.
- Deploy: long-running multi-service image builds should happen in GitHub
  PR/CI and publish immutable container tags; the VPS/Coolify deploy should pull
  and run those prebuilt images instead of rebuilding every service on the VPS.
- Repository layout decision: use a same-repo workspace layout with the Vercel
  React Router app under `apps/web` and the VPS backend under `apps/api`.
  Shared packages should be introduced only for stable contracts/schemas.
- Follow-up infrastructure direction: continue the workspace monorepo with
  Turborepo orchestration, dedicated API/worker/Bull Board containers, GitHub
  PR-built images, and Coolify/VPS pull-only deploys. Detailed plan:
  [`../monorepo-turborepo-container-plan.md`](../monorepo-turborepo-container-plan.md).

Backend API, schema, queue, storage, and lifecycle details are defined in
[`v0.13-backend-contract.md`](./v0.13-backend-contract.md).

## Non-Goals

- Do not add broad public AI access in v0.13.
- Do not expose provider keys to Vercel client code or browser code.
- Do not build full server-side project sync as part of the first AI slice.
- Do not build subscriptions, billing UI, galleries, community presets, or
  public project sharing in v0.13.
- Do not add advanced inpainting, fine-tuning, batch generation, or complex
  multi-provider routing before the minimal job and asset model is stable.

## Acceptance Criteria

- A logged-in, AI-enabled user can request an image generation from Artifact.
- A non-authenticated or non-enabled user cannot create a generation job.
- A user cannot exceed the configured monthly generation quota.
- Rate limiting protects the generation endpoint from accidental repeats and
  abuse.
- Generation jobs are durable and have visible states: queued, running,
  succeeded, failed, and cancelled or expired where supported.
- Provider secrets stay server-side on the VPS.
- Successful generation creates an asset record and a retrievable image file.
- The generated asset can be inserted into the current document and rendered by
  the existing renderer/export path.
- `.artifact.json` export/share behavior for generated assets is defined before
  the feature is considered complete.
- Failures are recoverable enough for a private alpha: the user sees an error,
  quota is not incorrectly consumed, and failed jobs can be inspected.

Alpha merge decision gate:

The first private alpha can merge before every v0.13 release-roadmap item is
complete, but only if the account-gated AI Image workflow is reliable enough to
test without losing local work, hiding failures, or creating unbounded provider
spend.

Merge blockers:

1. [ ] Current reliability batch is committed.
   - Includes generated-image history, selected-node loading overlay,
     import-failure terminal state, React Flow measurement retention, and
     `artifact-asset://...` preview preload fixes.
2. [ ] Full validation passes after the reliability commit.
   - Required: `npm run format:check`, `npm run lint`, `npm run typecheck`,
     `npm run typecheck:api`, `npm run test`, focused AI Image Playwright
     coverage, and `npm run build`.
3. [ ] Local VPS-shaped QA passes with real services.
   - Required path: Clerk login, enabled alpha account, Postgres, Redis,
     BullMQ worker, Bull Board, local generated-file storage, and OpenAI
     provider.
   - Required flow: first generation, second/third generation in the same node,

   - Recovery can be intentionally small for alpha: retry failed generation,

### 1. Product And Provider Research

Parallelizable: yes. Can run while backend scaffolding is prepared.

- Compare OpenAI GPT Image and xAI image generation for quality, latency, cost,
  resolution, image-to-image support, editing support, commercial-use terms,
  safety constraints, and API ergonomics.
- Decide the first provider/model defaults and which settings the UI should
  expose.
- Define Artifact's server-side prompt policy for the private alpha.
- Decide whether xAI is exposed as a user-selectable provider or an admin-only
  override in the first version.

Exit criteria:

- Provider defaults are documented.
- Prompt/settings surface is small enough for the first Image Generation node.
- Known provider risks and cost assumptions are written down.

### 2. Auth And Entitlements

Parallelizable: partly. Can start before the worker, but the API contract should
settle before frontend wiring.

- Choose auth provider or reuse existing infrastructure.
- Add login/registration route or external auth flow.
- Add user records with `ai_enabled`, role, or plus status.
- Define how the Vercel app authenticates to the VPS API.
- Add server-side checks that block generation for anonymous or non-enabled
  users.

Exit criteria:

- VPS API can reliably identify the current user.
- AI access can be enabled for only the intended private alpha users.

### 3. Generation API, Queue, And Worker

Parallelizable: partly. Worker can be built beside storage once the job schema is
defined.

- Add `POST /api/ai/generations` to create a job.
- Add `GET /api/ai/generations/:id` to read job status.
- Add cancellation endpoint if the provider and queue model support it cleanly.
- Add BullMQ queue and Redis connection.
- Add worker process with provider adapters, timeout handling, retry policy, and
  dead-letter/final-failure behavior.
- Add idempotency keys so double clicks do not create duplicate paid jobs.

Exit criteria:

- A test job can move from queued to succeeded or failed without blocking the
  request thread.
- Duplicate requests can be safely ignored or mapped to the original job.

### 4. Quotas, Rate Limits, And Cost Controls

Parallelizable: yes. Can run alongside worker implementation after auth exists.

- Add per-user monthly generation quota, initially something like 10
  generations/month.
- Add per-IP and per-user rate limits.
- Add one-active-generation-per-user concurrency guard for the private alpha.
- Track provider, model, estimated cost, and usage metadata where available.
- Decide whether failed provider calls consume quota.

Exit criteria:

- A user cannot create unbounded jobs.
- Admin can inspect usage and understand approximate monthly cost.

### 5. Asset Storage And Serialization

Parallelizable: partly. Can run beside frontend node work after the asset API is
sketched.

- Decide first storage backend: local VPS disk, S3-compatible object storage, or
  another existing file service.
- Store generated image metadata in DB and bytes in storage.
- Define generated asset metadata: provider, model, prompt, negative prompt if
  used, seed/settings where available, source asset references, creation time,
  license/commercial-use note, and storage reference.
- Define how generated assets hydrate into local IndexedDB for editing.
- Define `.artifact.json` export behavior for generated assets: generated
  outputs are normal local image assets after import, so save/share follows the
  existing image hydration path.
- Keep job state, quota/cost records, raw provider responses, and generated
  asset payloads outside `CanvasDocument`; the editor document should receive a
  normal image `src` reference once the generated asset is ready.

Exit criteria:

- Generated image files survive reload.
- Generated assets can be used by the editor like imported images.
- Project portability behavior is documented.

### 6. Frontend Image Generation Node

Parallelizable: partly. UI shell can start early, but job submission waits on the
API contract.

- Add a minimal Image Generation node/control surface.
- Include prompt, provider or mode, aspect ratio, quality/style preset, generate
  action, progress/status, quota messaging, and error display.
- Poll job status or subscribe to server updates.
- Insert successful output as a normal image asset/source in the document graph.
- Keep generated asset state serializable; do not store provider responses,
  blobs, DOM objects, or decoded images in `CanvasDocument`.

Current alpha scope:

- The add menu exposes `AI Image`, but it creates a normal image layer node named
  `AI Image` instead of introducing a new serialized layer kind.
- Image-node properties show the account-gated generation panel. Anonymous or
  non-enabled users can keep using the editor, while the unavailable AI action
  explains why an account/access is required.
- Successful generation imports the completed backend asset into the existing
  local `artifact-asset://...` store, then updates the image layer `src`.
- The resulting image layer stores a compact `aiGeneration` provenance object
  with the prompt and generation identifiers. Prompt provenance is shown with the
  image/card. Successful generated variants are kept as a lightweight
  `aiGenerationHistory` list so the image node can show a count and step between
  previous/next outputs. Provider payloads, quota/cost records, and queue state
  remain UI/server concerns.

Exit criteria:

- The generated image appears in preview and export through existing renderer
  paths.
- The node can recover from failed or pending jobs without corrupting document
  state.

### 7. Observability, Admin, And Operations

Parallelizable: yes. Can begin once job records exist.

- Add structured logs for job creation, provider request, provider response,
  storage write, and failure states.
- Add basic admin visibility for jobs, users, quota usage, and provider errors.
- Add cleanup for orphaned files and expired failed outputs.
- Add DB backup plan and storage backup decision.
- Add deployment/runbook notes for VPS API, worker, Redis, DB, and storage.
- Add CI-built container deploys for VPS/Coolify: GitHub PR/CI builds API,
  worker, and admin/Bull Board images, pushes immutable tags to a registry, and
  deploys those built images so the VPS does not spend preview-deploy time on
  duplicate `npm ci` / application builds.

Exit criteria:

- Failed jobs can be diagnosed without guessing.
- The private alpha can be operated without manually querying every table.
- Preview and production deploys pull known image tags, and image build failures
  are visible in CI before the VPS deploy step starts.

## Manual QA

- Anonymous user cannot generate.
- Logged-in user without AI access cannot generate.
- AI-enabled user can generate one image and use it in preview/export.
- Quota exhaustion blocks new jobs with clear messaging.
- Double-clicking generate does not create duplicate paid jobs.
- Provider failure shows a recoverable error.
- Failed completed-asset import leaves the AI Image node in a terminal failed
  state, not infinite importing/generating.
- Generating multiple images in the same AI Image node preserves all successful
  variants and allows previous/next traversal after reload.
- Node preview, gallery preview, and export paths resolve generated local
  `artifact-asset://...` references instead of loading them as browser URLs.
- Reloading the app preserves access to the generated asset.
- Exported project behavior matches the documented generated-asset portability
  decision.
