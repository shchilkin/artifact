# v0.31 Code Quality And Fallow Integration

## Purpose

v0.31 is a code-quality release. It should make Artifact easier to maintain
after the v0.30 design-system baseline by adding Fallow to local, CI, and agent
workflows, then using the reports to create a trustworthy cleanup backlog.

This release is not a product-feature release. Its job is to improve codebase
intelligence, changed-code review, cleanup confidence, and release hygiene
without changing editor semantics.

## Product Direction

Artifact is growing into a graph-heavy creative editor with renderer, storage,
AI, local projects, docs, and design-system surfaces. v0.31 should reduce
maintenance risk before the next product slice by answering practical questions:

- Which files, exports, types, and dependencies look unused?
- Which components or helpers are duplicated enough to deserve consolidation?
- Which functions and files are complexity hotspots?
- Which dependency or workspace boundaries need clearer rules?
- Which changed files introduce new quality risk?
- Which Fallow findings are actionable now, and which need suppression or
  future validation?

## Non-Goals

- No landing redesign.
- No Showcase / How-to split.
- No command palette or Add Library feature work.
- No server-backed sharing, account sync, or backend product feature.
- No broad UI rewrite.
- No renderer, graph traversal, export, persistence, document-schema, package
  export, AI scope, or font-policy semantic changes unless a Fallow finding
  exposes a small, verified bug fix.
- No strict CI failure gate until the baseline is understood and documented.
- No automatic deletion from Fallow findings without trace validation and the
  normal test gate.

## Workstreams

### 1. Fallow Baseline

- [ ] Install or configure Fallow in the project workflow without enabling
  telemetry.
- [ ] Add a read-only local command for dead code, duplication, health,
  complexity, circular dependencies, and dependency placement.
- [ ] Save or document the first baseline in a stable format suitable for
  comparing future runs.
- [ ] Classify findings into actionable cleanup, needs-trace, accepted
  suppression, and false-positive categories.

### 2. Agent Workflow

- [ ] Document exact agent-safe commands, including JSON output, `--quiet`,
  `--explain`, and `|| true` handling.
- [ ] Add rules for when agents may run Fallow during code review, release
  prep, refactors, or cleanup passes.
- [ ] Require `--dry-run` and trace validation before any auto-fix or deletion.
- [ ] Add guidance for interpreting unused exports, dependency findings,
  duplication fingerprints, complexity hotspots, and changed-code audit.

### 3. CI Soft Gate

- [ ] Add a CI job or script that can run Fallow in report-only mode.
- [ ] Keep the initial gate non-blocking or warning-only until the baseline is
  accepted.
- [ ] Scope changed-code audit to the PR base branch when possible.
- [ ] Document what would need to be true before Fallow becomes a stricter
  release or PR gate.

### 4. Cleanup Backlog

- [ ] Use Fallow output to create a prioritized cleanup list.
- [ ] Separate safe cleanup from product-risky refactors.
- [ ] Identify duplicated UI primitives or helper logic that should move into
  the v0.30 design-system foundation.
- [ ] Identify dead-code candidates that need browser/render validation before
  removal.

### 5. Validation And Release Gate

- [ ] Update `docs/testing.md`, `docs/agent-skills.md`, and any relevant agent
  instructions with the Fallow workflow.
- [ ] Update `docs/production-readiness.md` with v0.31 release prep and whether
  Fallow was report-only or gated.
- [ ] Create `docs/releases/v0.31.0.md` from the release template during release
  prep.
- [ ] Run the normal release gate.

## Acceptance Criteria

- Fallow has a documented local workflow for code-quality reports.
- Fallow has a documented agent workflow that avoids telemetry, avoids
  destructive fixes without dry-run review, and treats JSON output correctly.
- CI can run Fallow in report-only or soft-gate mode without blocking on the
  initial baseline.
- The first baseline is interpreted and turned into a concrete cleanup backlog.
- Any cleanup shipped inside v0.31 is small, verified, and covered by the
  lowest useful tests.
- Product-facing roadmap work remains deferred out of v0.31.

## Validation

- [ ] `npm run check`
- [ ] `npm run build`
- [ ] `npm run test:browser`
- [ ] Fallow local baseline command
- [ ] Fallow changed-code audit command
- [ ] `npm run perf:node-editor` only if cleanup touches React Flow,
  node-editor rendering, high-frequency gestures, thumbnail scheduling, or other
  performance-sensitive editor paths.

## Suggested Implementation Order

1. Confirm Fallow CLI availability and choose project command names.
2. Run read-only local reports and capture the first baseline.
3. Document agent-safe command usage before acting on findings.
4. Add CI report-only or soft-gate wiring.
5. Convert findings into a cleanup backlog.
6. Apply only the safest cleanup items with focused tests.
7. Re-run normal release gate and Fallow reports before release.
