# v0.40 Shader Authoring And AI Shader Effects

Status: release-candidate prep complete on the stacked shader branch. Package
metadata, release notes, architecture docs, browser coverage, and the local
release gate are prepared before promotion to `development` and `main`.

## Purpose

v0.40 turns shader exploration into a bounded authoring workflow. Users can add
curated procedural shader fills, process connected artwork with shader effects,
write GLSL in a Code Shader, or ask AI for an editable input-dependent shader
without changing graph traversal or introducing arbitrary JavaScript execution.

The release thesis is: shader nodes can create textures or transform existing
artwork with explicit roles, editable controls, transparent failure behavior,
and preview/export parity.

## Product Direction

- Keep every shader node single-purpose with one explicit `fill` or `effect`
  role.
- Keep Code Shader and AI Shader on one serializable Shader Definition / Shader
  Instance runtime while preserving curated preset renderers.
- Treat AI Shader as an input-dependent effect: it requires a connected source,
  validates generated GLSL in the browser, and keeps the previous accepted
  result until a replacement passes.
- Make local fallback an explicit labeled choice after an AI failure, never a
  silent substitution.
- Keep empty, loading, validation, repair, failure, fallback, accepted, and
  refinement states understandable on both node and inspector surfaces.

## Non-Goals

- No shader animation, timeline node, video export, or per-frame document
  mutation.
- No reusable custom shader library, Save As Preset workflow, definition
  version history, marketplace, or community sharing.
- No arbitrary JavaScript, network access, DOM access, or user-provided compute
  outside the constrained GLSL fragment contract.
- No material-lighting, 3D scene, or export controls inside shader nodes.
- No AI billing UI, monetary cost estimates, or generation history browser.

## Workstreams

- [x] Add Shader Fill and Shader Effect as explicit connection-independent
  roles.
- [x] Let shader fills create texture sources and shader effects process one

- [x] Preserve graph preview, thumbnail, output, and export parity.
- [x] Keep missing or invalid effect inputs transparent instead of substituting

### 2. Code Shader Authoring

- [x] Persist serializable Shader Definitions, Shader Instances, and dynamic

- [x] Contain WebGL failures, release disposable contexts, and render failed

- [x] Show only controls whose uniforms participate in the custom shader.

### 3. AI Shader Lifecycle

- [x] Generate editable source-aware GLSL and manifest controls through the

- [x] Offer a clearly labeled deterministic local draft only after recoverable

- [x] Support refinement from an accepted request with durable provenance and

## Acceptance Criteria

- A Shader Fill can feed artwork or a material texture input without requiring
  an upstream image.
- A Shader Effect requires a backdrop, transforms that source, and renders
  transparently when the source or runtime is unavailable.
- An empty Code Shader is transparent; invalid GLSL reports a compilation error
  without destabilizing the editor or output path.
- AI generation never silently falls back, never replaces a working shader
  before browser acceptance, and clearly marks local drafts.
- Generated properties are editable and visibly affect accepted shader output.
- Accepted AI shaders can be refined while preserving request provenance.
- Preview, node thumbnail, graph output, and export use the shared shader
  runtime with covered orientation/parity behavior.
- Release notes, roadmap, production readiness, and architecture docs describe
  the shipped model and its deferred animation/preset work.
