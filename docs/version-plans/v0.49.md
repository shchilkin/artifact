# v0.49 AI-Assisted Creation

Status: planned. The domain model and MVP boundary are accepted; implementation
is tracked in GitHub milestone
[`v0.49 AI-Assisted Creation`](https://github.com/shchilkin/artifact/milestone/1).

## Purpose

v0.49 makes Artifact usable through conversation without creating a separate,
flattened creative system. A signed-in user can describe a direction or concrete
design, receive an editable Artifact document, revise it through previewed
changes, and continue in Chat, Layers, or Nodes.

The release thesis is: Artifact can turn a creative request into an editable
Creative Direction or Composition and safely continue that work through either
conversation or the existing editors.

The detailed product, domain, persistence, and interaction contract lives in
[`../ai-assisted-creation.md`](../ai-assisted-creation.md).

## Product Direction

- Treat Chat, Layers, and Nodes as three peer modes of one Project.
- Keep Chat Mode full-screen and focused on creation rather than embedding a
  reduced editor beside the conversation.
- Add the Context Assistant only as an explicitly invoked, scope-bound editor
  tool. It must not analyze work or show suggestions proactively.
- Produce editable `CanvasDocument` payloads for both Creative Directions and
  Compositions; keep intent, derivation, revisions, Chats, and Runs in
  Project-owned server records outside the renderer model.
- Keep generated images as replaceable raster sources with editable placement,
  masks, effects, provenance, and regeneration prompts.
- Preview every assistant edit as a revision-bound Change Set before Apply.
- Count one successful Run as one Generation while recording each internal
  provider call as Provider Usage.
- Route providers by `Fast`, `Best quality`, or `No image generation`; use Grok
  Imagine as an eligible fast image provider without making provider choice the
  primary workflow.

## Non-Goals

- No anonymous or local-only Chat history.
- No autonomous critique, background analysis, or unsolicited suggestions.
- No pixel-level decomposition of generated raster images.
- No real-time multi-user collaboration.
- No unlimited image variants inside one Run.
- No replacement renderer, graph model, or editor-only document format.
- No raw JSON or document command stream as the primary Change Preview UI.
- No billing, checkout, or change to the v0.41 Account Tier product policy.

## Domain And Architecture

The durable ownership shape is:

```text
Project
|- Chats -> Messages -> Runs
|- Context Threads
|- Documents -> Revisions
`- Assets
```

`CanvasDocument` remains the serialized creative payload and the canonical input
to preview, thumbnails, editor, and export. Project document records own intent,
current revision, and derivation metadata. Runs own asynchronous orchestration;
provider jobs remain lower-level execution details.

AI edits are represented by typed Change Sets against a `baseRevisionId`.
Applying a proposal creates one durable revision and one undoable creative
action. Independent changes may rebase; conflicts, destructive graph changes,
locked deletion, and invalid connections must be surfaced rather than applied
silently.

## Acceptance Criteria

- A signed-in user can create a Chat with a stable server ID and reopen its
  history in another browser session.
- One Project can keep multiple Creative Direction and Composition documents,
  their revisions, assets, and derivation relationships.
- A no-image request creates an editable Composition through a typed setup and
  canonical preview.
- An image-backed request routes through an eligible provider, persists the
  asset, and opens the result as a replaceable image source in the editor.
- A Creative Direction can produce a linked Composition without overwriting the
  source document.
- Runs survive reloads and support accurate progress, cancellation, and failed
  stage retry.
- One successful Run commits exactly one Generation; internal calls remain
  separately auditable Provider Usage.
- Every assistant edit is previewed as an editable Change Set before it can
  create a revision.
- Stale proposals never overwrite conflicting manual edits silently.
- Context Assistant work starts only after explicit invocation and cannot write
  outside its accepted scope.
- `View in editor` and return to Chat preserve Project, document, Chat, Run, and
  conversation context.
- Preview, thumbnails, Layers, Nodes, and export continue to use the canonical
  Artifact renderer for the same `CanvasDocument` state.

## Validation

Focused implementation coverage must include:

- repository and API tests for owner-scoped Projects, documents, revisions,
  Chats, Context Threads, Runs, Change Sets, and assets;
- pure tests for typed document operations, graph validation, destructive-change
  classification, rebase, and conflicts;
- render tests for transient proposals and applied revisions through the
  canonical renderer;
- browser tests for Chat history, waiting states, setup-only creation, generated
  images, derivation, revision Apply/Discard, conflicts, and both assistant
  scopes;
- account tests for reservation, success, failure, retry, Provider Usage, active
  operation limits, and Safety Budget behavior;
- responsive and accessibility QA for Chat, Change Preview, editor rails, and
  drawers.

Required release gate:

```bash
npm run check
npm run build
npm run test:browser:release
```

The final release run must include one controlled real provider smoke before the
exact verified revision is promoted and tagged `v0.49.0`.

## Delivery Sequence

1. Land Project documents and revisions without breaking existing cloud saves.
2. Land authenticated Chat history and the setup-only vertical slice.
3. Make Runs durable and connect them to v0.41 accounting.
4. Add adaptive image generation and Creative Direction derivation.
5. Add revision-bound editing and interactive Change Preview.
6. Reuse that engine for whole-document and scoped Context Assistant flows.
7. Run the complete release gate, provider smoke, release notes, and exact-SHA
   promotion.

Each issue should fit one fresh implementation context and remain demoable on
its own. v0.49 is complete only when #136 closes; partial Chat or assistant
surfaces are not release-complete by themselves.
