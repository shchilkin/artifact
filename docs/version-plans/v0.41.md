# v0.41 Account Tiers And Backoffice

Status: discovery complete and implementation ready. No v0.41 code has shipped.

## Purpose

v0.41 replaces ad hoc AI access flags with explicit Free, Creator, and Founder
Account Tiers, makes Generation allowances safe under concurrency, records
provider cost, and gives one Admin a separate operational backoffice.

The release thesis is: Artifact can grant and operate AI access predictably,
attribute provider cost to accounts, and stop runaway spend without exposing
users' creative work to the backoffice.

## Product Direction

- Keep Account Tier independent from User Role. Founder does not imply Admin.
- Keep Tier Policy fixed in server code for the first release: Free has no
  provider-backed AI, Creator has 20 successful Generations per UTC month, and
  Founder has no finite product allowance.
- Count one usable user-initiated image create, shader create, or shader refine
  as one Generation. Failures and automatic shader repair do not consume an
  additional Generation.
- Attribute every provider call through append-only Usage Events without
  storing prompts, shader source, generated assets, or project documents.
- Run `apps/backoffice` as a separate React Router/Vite application and VPS
  service at `backoffice.artifact.shchilkin.dev`.
- Require both Cloudflare Access and Better Auth Admin authorization in
  production.

## Non-Goals

- No billing, checkout, subscription provider, upgrade purchase, or Creator
  access-request workflow.
- No configurable Tier Policy UI. Limits remain server policy until a later
  audited configuration release.
- No account suspension/reactivation UI and no role-management UI.
- No prompt, shader-code, generated-asset, or project-document inspection.
- No teams, organizations, seats, storage quotas, invoices, refunds, or tax
  reporting.
- No Save As Preset or shader library work.

## Domain And Policy

| Tier | Provider-backed AI | Monthly product allowance |
| --- | --- | --- |
| Free | unavailable | 0 Generations |
| Creator | available | 20 successful Generations |
| Founder | available | no finite product allowance |

All periods use UTC. Quota Grants add a positive allowance to one account for
one period and never roll over. Corrections use linked Quota Grant Reversals.
Tier changes take effect immediately and never reset current-month usage.

The initial global Safety Budget is USD 30 per UTC month, warning at USD 24 and
rejecting new AI operations at USD 30. Operational limits are one active AI
operation per account, ten starts per rolling minute per account, and two
concurrent provider calls per worker for Creator and Founder alike.

## Data Model

Implementation should introduce explicit, migration-owned records rather than
overloading `ai_enabled`, `plus_status`, or a mutable monthly counter:

- `account_access`: current tier, optimistic version, and update metadata for
  one account.
- `tier_assignments`: append-only before/after tier, reason, Admin, timestamp,
  and idempotency metadata.
- `quota_grants`: positive amount, UTC period, reason, Admin, and idempotency
  metadata.
- `quota_grant_reversals`: amount and reason linked to one grant.
- `ai_operations`: user action, feature, idempotency key, reservation period,
  and reserved/running/succeeded/failed/cancelled/expired lifecycle shared by
  image and shader flows.
- `ai_usage_events`: one provider call with provider/model/feature/status,
  request ID, normalized usage, integer micro-USD cost, and pricing version.
- `ai_usage_monthly`: durable aggregate for Generations, tokens, cost, status,
  provider, and model.
- `provider_reconciliations`: imported provider totals, difference from the
  internal ledger, and sync status.
- `admin_audit_events`: immutable administrative mutation history.

Detailed Usage Events are retained for 24 months. Monthly aggregates, Tier
Assignments, Quota Grants, reversals, and admin audit records are retained
indefinitely.

## Acceptance Criteria

- Free cannot start provider-backed AI; Creator can consume 20 successful
  Generations per UTC month; Founder has no finite product allowance but remains
  subject to operational limits and the Safety Budget.
- Parallel or retried requests cannot spend the same allowance or create
  duplicate grants.
- Failed provider work releases its Generation reservation; successful work
  commits exactly one; automatic repair adds Provider Usage but no Generation.
- Tier changes, grants, and reversals are immediate, audited, and do not reset
  usage or revoke existing results.
- Internal usage attributes provider/model/tokens/cost/status to accounts while
  excluding Creative Content, and daily provider reconciliation exposes any
  difference.
- New provider-backed work stops at USD 30 monthly estimated spend while the
  non-AI editor remains usable.
- A non-Admin cannot load backoffice data or mutate accounts even if the SPA or
  endpoint URL is known.
- The four backoffice views handle empty, loading, error, conflict, and normal
  states and remain usable with paginated data.
- The production backoffice is a separate VPS service behind Cloudflare Access
  and Better Auth Admin authorization.
- Legacy entitlement flags no longer affect access after migration.

## Validation

Focused implementation coverage:

```bash
npm run test -- accountAccess
npm run test -- quota
npm run test -- admin
npm run typecheck
npm run build
```

Required release gate:

```bash
npm run release:verify
npm run check
npm run build
npm run test:browser:release
npm run test:e2e:backoffice:run
```

The browser/E2E gate must cover Free denied, Admin assigns Creator, one real or
deterministic Generation consumes allowance, Quota Grant increases remaining
allowance, duplicate mutations remain idempotent, and non-Admin access is
denied. Run the account migration in dry-run mode before applying it.

Manual production QA includes a Cloudflare Access and Admin-auth smoke without
a paid provider call, followed by one controlled real OpenAI operation before
release promotion. Provider credentials and Creative Content must not appear in
browser responses, logs, screenshots, or backoffice records.

## Delivery Sequence

1. Land schema, repositories, domain calculations, and migration dry run.
2. Move existing AI endpoints onto shared access, reservation, and usage logic.
3. Land Safety Budget and provider reconciliation.
4. Land Admin bootstrap and `/api/admin/*` contracts.
5. Land the backoffice application one route at a time.
6. Land account-facing AI access states.
7. Deploy the isolated VPS service and Cloudflare Access, then run the release
   gate and controlled provider smoke.

Each step should be independently testable and keep existing AI behavior
available until the entitlement cutover migration is deliberately applied.
