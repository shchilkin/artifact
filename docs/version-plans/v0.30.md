# v0.30 Editor Visual Baseline And Design System

## Purpose

v0.30 should make future editor work safer by adding a small, deliberate visual
regression baseline and beginning the editor design system foundation. The
release should protect the surfaces that are hardest to reason about from unit
tests alone: Layers, Nodes, Add Library, previews, output context,
selected/hidden/locked states, and reused UI primitives.

This is not a new product surface release. It is a confidence release that
protects the editor and starts turning repeated UI chrome into one reusable
source of truth before the roadmap moves into larger product or infrastructure
work.

## Product Direction

The editor is the product proof. Users compose covers, posters, texture studies,
and images from text, images, noise, effects, layers, and nodes. v0.30 should
verify that the visible editor still communicates that model after code changes:

- Layers are readable as the fast stack workflow.
- Nodes are readable as the explicit graph workflow.
- Add Library remains legible as the way to add sources, effects, and utilities.
- Preview and output surfaces remain nonblank and visually aligned with the
  selected document.
- Guardrail states such as selected, hidden, locked, off-output-path, and
  no-input remain visible.
- Shared primitives and tokens become visible in a style-guide route so future
  editor changes reuse the same controls instead of inventing local geometry.

## Non-Goals

- No landing redesign.
- No Showcase / How-to split.
- No broad new editor feature surface. Small editor usability fixes discovered
  during real project testing are allowed when they stay model-first and receive
  focused logic/render/browser coverage.
- No renderer, graph traversal, export, persistence, or document-schema
  semantics changes.
- No broad snapshot sweep over every route. The baseline should be curated and
  low-noise.
- No visual tests that depend on live network fonts, AI generation, or
  nondeterministic image loading.
- No full editor rewrite. Migrate the lowest-risk shared primitives first and
  leave feature-specific surfaces in place until the primitive contract is
  visible and tested.
- No new external component library. The project already has Tailwind,
  `tailwind-merge`, `radix-ui`, and source-owned UI wrappers; v0.30 should not
  add Storybook, Chromatic, or another visual system by default.

  primitives while preserving feature-local row geometry and drag/add behavior.
- Product-heavy surfaces stay out of this migration pass so v0.30 can close as
  a baseline release, not an editor rewrite.

## Acceptance Criteria

- v0.30 has a curated editor visual regression baseline that protects the
  highest-risk editor states without producing a noisy snapshot suite.
- Layers, Nodes, Add Library, selected/hidden/locked states, output context, and
  preview surfaces are represented by focused browser coverage.
- Snapshot or screenshot assertions use deterministic local fixtures and do not
  depend on live network assets.
- Existing pure logic, render, and browser tests continue to cover behavior;
  visual coverage does not replace lower-level tests.
- `docs/style-guide.md` exists as the project UI style guide, and
  `docs/editor-design-system.md` extends it with editor-specific token
  families, primitive tiers, Radix/shadcn boundaries, style-guide route
  expectations, and migration order.
- A style-guide route exists for visible UI components, design primitives, and
  key states, with deterministic examples and focused browser coverage.
- Shared base primitives start replacing repeated local button/input/search/menu
  styling without changing editor semantics.
- Effect nodes can vary stochastic output through local seed offsets while
  preserving preview/export determinism.
- Single-node drag shows alignment guides and snaps near peer edges/centers
  without changing graph traversal or serialization semantics.
- Radix/shadcn usage stays source-owned under `apps/web/app/components/ui/*`
  and uses Artifact tokens rather than imported default visual language.
- Editor/render/export semantics remain unchanged.
- Landing and Showcase / How-to work remain deferred and do not block v0.30.

## Suggested Implementation Order

1. Inventory current browser coverage and name the missing visual states.
2. Add deterministic fixture helpers before adding snapshots.
3. Add Layers baseline first because it has fewer moving parts.
4. Add Nodes baseline second, keeping graph documents small and stable.
5. Add preview/output parity checks only where they catch real regressions.
6. Inventory repeated UI primitives and local style values.
7. Define initial tokens and style-guide route structure.
8. Add the style-guide route and focused route coverage.
9. Migrate low-risk base primitives before product-specific editor primitives.
10. Update testing docs and release notes after the baseline and design-system
    foundation exist.

## Tools And Libraries

No new dependency is required for the v0.30 foundation by default.

Use what is already present:

- **Tailwind** for simple layout and low-state wrappers.
- **CSS variables and feature CSS** for Artifact's visual contract.
- **`tailwind-merge` / `cn()`** for class composition in shared primitives.
- **`radix-ui`** for accessible mechanics in dialogs, sheets, tabs, menus,
  popovers, tooltips, switches, selects, and other keyboard/focus-heavy
  controls.
- **Playwright** for deterministic style-guide and editor visual checks.

Optional later, not required for v0.30:

- **shadcn CLI** only to copy source-owned primitive implementations when a
  Radix behavior family is needed. If used, adapt the copied source to Artifact
  tokens immediately.
- **Storybook** only after the internal style-guide route becomes too large or
  needs external design review.
- **Chromatic or another visual SaaS** only after golden screenshots prove
  stable across supported browsers.
