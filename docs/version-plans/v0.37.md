# v0.37 3D Material Nodes

Status: release-candidate complete. The implementation target and release gate
are recorded in `docs/releases/v0.37.0.md` and
`docs/production-readiness.md`.

## Purpose

v0.37 extends Artifact's existing 3D primitive/model/scene workflow with one
first-class PBR material node. A user can create a material, preview it as its
own graph node, connect it to a 3D primitive or 3D Scene material input, tune
lighting and environment, then compose the rendered 3D result with the text,
mask, image, and effect tools Artifact already has.

The motivating reference is editorial foil typography, but the release is not a
new text-mask system or a flat gold effect. The missing piece was the material
layer of the 3D story: a graph material input that can drive metallic foil,
chrome, brushed metal, paper, fabric, plastic, pearl, and other physically
inspired surfaces through the same preview/export path as the rest of the
document.

## Completed Scope

- Added serializable `GraphMaterialNode` records under `CanvasGraph` with
  JSON-safe PBR controls for base/accent color, metalness, roughness,
  clearcoat, relief, grain, anisotropy, and texture-map references.
- Added explicit material graph ports so material edges connect to 3D
  primitive/model scene inputs without overloading pixel-flow `in`, `bg`,
  `a`, or `b` ports.
- Added texture-map input ports for albedo, roughness, metalness, normal, and
  alpha. When a map is connected, the matching direct material control is
  disabled in the inspector so the graph source owns that field.
- Added one PBR Material Add Library entry. Preset values remain available as
  starting recipes inside that single material node; they are not a separate
  material-library product surface.
- Added material node previews and render signatures so thumbnail invalidation
  responds to material controls, texture sources, material edges, scene fields,
  primitive/model fields, and camera changes.
- Wired connected PBR materials into live 3D primitive previews, 3D Scene
  thumbnails, model scene rendering, graph output previews, and raster export
  through shared Three.js material helpers.
- Added explicit Environment Map graph nodes that can use imported HDR/EXR
  assets or graph-rendered 2:1 source pixels, then feed 3D Scene lighting and
  reflections rather than acting only as a backdrop image.
- Kept generated maps, Three.js materials, textures, render targets, canvases,
  and WebGL objects outside `CanvasDocument`; only serializable source
  references and numeric/string controls persist.
- Preserved primitive/model camera state while material and environment inputs
  change, including the existing graph `primitiveViewStates` ownership model.
- Kept rendered 3D scene output as normal graph pixels downstream, so existing
  merge, color, effect, text, mask, imported-font, and export workflows remain
  the composition layer.
- Added multiple-resource drag/drop support so a user can bring several
  compatible material/environment/image resources into the graph in one drop.
- Added focused unit, render, and browser coverage for material node defaults,
  cloning, normalization, graph insertion, material-port validation,
  texture-map asset references, render signatures, connected material rendering,
  material-node Add Library search, and 3D Scene material input smoke.

## Architecture Notes

Material input is intentionally not a normal pixel-flow edge. The implementation
uses dedicated material and material-texture ports so graph traversal can treat
materials as render inputs to 3D nodes instead of normal branch compositing.

The first release keeps material nodes graph-only. Layers expose 3D material
settings as controls on primitive/model/scene surfaces, while Nodes exposes
the PBR Material and Environment Map nodes explicitly.

Environment maps are likewise graph inputs to the 3D scene pipeline. A rendered
2:1 source can become an environment input, while imported environment assets
continue to persist through the existing local asset path.

## Non-Goals

- No broad new text-mask system. Text and mask composition reuses existing
  text, alpha, graph, and effect workflows.
- No flat "gold effect" that bypasses the 3D scene.
- No full 3D scene editor, mesh authoring, rigging, ray tracing, WebGPU
  migration, or timeline animation.
- No external HDRI/material marketplace.
- No broad font policy changes.
- No renderer shortcut where preview and export use different material
  semantics.
- No standalone material-library product surface beyond the one PBR node and
  its starter values.

## Acceptance Criteria

- A user can add a PBR Material node, preview it, and connect it to a 3D
  primitive/model scene material input.
- Connected material changes update live primitive/model scene preview, graph
  output preview, node thumbnails, and raster export through the same scene
  recipe.
- A 3D material scene can render full-frame when the artwork needs material to
  occupy the whole image.
- Existing primitive camera state, model scene framing, full-frame primitive
  rendering, text nodes, imported fonts, masks, effects, stack rendering, graph
  traversal, and export remain compatible.
- Material-node documents serialize through local persistence, project
  save/open, `.artifact.json` export/import, and URL/share import normalization
  without storing WebGL, DOM, texture, or canvas objects in the document.
- Undo/redo covers material creation, material control edits, graph connection
  changes, scene/light edits, and resource insertion through the existing
  document history flow.
- Missing WebGL or failed asset decode produces the existing visible fallback
  frame instead of a blank graph/export result.

## Follow-Ups

- Deeper authoring controls for map scale, map rotation, and per-map channel
  packing remain future material-workflow polish.
- Broader Firefox/WebKit WebGL material coverage remains future coverage work;
  the heavy 3D material smoke stays Chromium-first for this release.
- Larger material example packs and external HDRI/material marketplace flows
  remain outside v0.37.
- Startup node-editor long tasks from WebGL/thumbnail work remain a
  performance follow-up if the release benchmark keeps the existing pattern:
  interactive drag/slider/pan is healthy while initial load still has spikes.
