# v0.39 Account Cloud Saves And Production Auth

Status: release-candidate prep in progress after PRs #83, #84, #85, #86, #87,
#88, #89, #90, #92, #93, and #95 landed in `development`. Package metadata,
release notes, roadmap updates, and the release gate are prepared in the v0.39
release-prep branch before promotion to `main`, tagging, and GitHub Release
publication.

## Purpose

v0.39 turns Artifact's local Projects model into an account-backed workflow
while preserving local-first editing. It replaces Clerk with Better Auth, adds
cloud project saves, splits large project assets from the project manifest, and
hardens the Coolify/Vercel production path that serves the API.

The release thesis is: users can create an account, recover access, save local
projects to the cloud, and understand sync state without losing the fast local
editor workflow.

## Product Direction

Projects should feel local-first but no longer trapped in one browser. v0.39
pulls the account, API, and storage work into one bounded release:

- Replace the external Clerk dependency with a self-owned Better Auth account
  flow that works with the existing API and production database.
- Let local projects become cloud projects without forcing the editor document
  itself to carry server-only state.
- Split large cloud projects into a compact project manifest plus separately
  uploaded image/font/model/environment assets so oversized `.artifact` files
  have a clear path to sync.
- Make project sync states readable in the Projects UI: local-only, synced,
  syncing, failed, and too-large states should each explain what the user can
  do next.
- Make production deploys safer by running migrations during API startup,
  exposing backend healthchecks, allowing Vercel preview origins, and fixing
  Redis persistence behavior in Coolify.
- Keep editor rendering, graph traversal, package export, and document schema
  behavior stable while the account-backed project layer ships.

## Non-Goals

- No public share-link permission model, link tokens, or collaboration
  workflow. Cloud projects are account-owned private saves.
- No S3-compatible object storage migration. v0.39 keeps the initial
  local-volume cloud asset storage path.
- No cross-project asset deduplication, quota UI, cleanup jobs, or storage
  billing model.
- No project versions, restore/compare, autosave history, or team ownership.
- No broad redesign of the authentication modal beyond the recovery/resend
  polish needed for the new account flow.
- No renderer/export semantic changes, graph traversal changes, font-policy
  changes, or AI generation scope changes.

## Acceptance Criteria

- Users can create an account, sign in, sign out, recover a password, and load
  the current session through the API-backed Better Auth flow.
- A local project can be saved to the cloud and later loaded without dropping
  local browser project behavior.
- Large projects can sync through separately uploaded cloud assets rather than
  failing only because the project manifest is too large.
- Oversized or failed sync attempts show explicit UI states and retry affordance
  where retry is meaningful.
- Production API deploys can boot with migrations, healthchecks, Redis, and
  preview origins configured.
- Cloud projects remain private to the authenticated owner. Public sharing and
  ownership transfer are explicitly deferred.
- Existing editor rendering, export, graph traversal, and document persistence
  tests remain green.
- `docs/roadmap.md` points to this plan as the v0.39 release scope, and release
  prep creates `docs/releases/v0.39.0.md` from `docs/release-template.md`.

## Validation

Focused validation during implementation:

```bash
npm run test -- ProjectsPanel
npm run typecheck
npm run format:check
npm run build
```

Release gate before tagging v0.39.0:

```bash
npm run release:verify
npm run check
npm run build
npm run test:browser
```

`npm run perf:node-editor` is not required for v0.39 release prep because the
release changes account-backed persistence, API deploy behavior, project asset
sync, and Projects/status UI rather than React Flow gesture paths, graph
traversal, thumbnail scheduling, or renderer/export behavior.

Manual QA before release prep is complete:

- Create an account, sign in, sign out, and sign back in.
- Trigger password recovery and confirm the reset email is delivered.
- Save a local project to the cloud, reload it, and confirm Projects status
  communicates the sync state.
- Load a large asset-backed project and confirm the project opens after client
  asset reassembly.
- Confirm `/api/health` stays green after production migration startup.

## Follow-Ups

- Public share links, sharing permissions, link tokens, ownership transfer, and
  team/workspace models need a dedicated security and product design pass.
- S3-compatible object storage, quota/usage UI, cleanup jobs, and asset
  deduplication remain future cloud-storage work.
- Project autosave history, project versions, restore/compare, and explicit
  cloud/local conflict handling remain separate product releases.
- Broader account settings, profile UI, and notification preferences remain
  future polish after the account core is proven.
